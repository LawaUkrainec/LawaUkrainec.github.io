Overview

You play as Mr. Skellington, a skeleton who wants nothing more than health in his old age. Luckily, milk can help create strong bones and he happened to find a cave full of it! Your goal is to collect all of the milk cartons in the cave, hopping from platform to platform and timing your jump onto the elevator to reach cartons on higher levels.

Process

This game began a sprite animation script and a desire to create a simple 2D platformer with collectables. I like to make things look good, so my first step was taking care of all the visual content. I found basic cave textures and a free-to-use sprite sheet for Mr. Skellington, then used Photoshop to make them more cohesive. I also made the collectables and all of the UI screens using Photoshop.

Then I moved on to the code itself. Since the animation sequence had been provided for me, I began with a very important part of any game: colliders. I wrote separate collider classes for each side which all inherited from a base collider class. This way, I could control exactly what happens when the player interacts with anything from any side. Of course, all of my colliders had a toggleable draw state for easier debugging. This led me to the platforms, which are essentially a list of 4 colliders with a texture. I also added a moving platform for an extra challenge.

Next up was Mr. Skellington himself. He has 3 enum states that control the sprite animations and methods to handle all possible movements. This includes basic movement, such as horizontal/vertical or both, as well as jumping, landing, and simply standing on things. This script keeps track of his position and internal bounding box for collision processing, but doesn't react to the collision. I felt that was better suited for the collider scripts.

Finally, I wrote the meat of the game. I decided to use enums to control the overall game state since they can be easily modified if I wanted to add more levels. I set up all the basics, such as the game window and lists to keep track of colliders, platforms, and the collectables. The lists made it easy to apply logic to everything at the same time, like initializing, updating/drawing, and monitoring the collectables to check for a win.

Accomplishments

Mr. Skellington was the first complete game that I made in Monogame, which is very code-heavy. This meant that I had to figure out how to create objects that interact with each other, or in other words, collide. I realized that a collider is just a box around the sprite's texture, and from there I wrote a base collider class that keeps track of its position and dimensions. Then I made separate child classes to control collisions from every direction. This might not seem very impressive, but this had me stumped for a while and I was extremely proud of myself when I figured it out.

Unlike Unity, Monogame doesn't use different scenes to transition between stages of the game, such as Start, Playing, and End. I solved this by using enums to store the current stage, pairing it with switch statements to emulate separate game stages. Like having different user inputs for the Start screen than the Playing phase, for example, and only drawing the congratulations panel after the player has collected everything.

The most challenging aspect of this game was the elevator platform. Though I'd already puzzled out colliders, I had a new problem: how do I make the player move with the elevator instead of staying stationary or clipping through it? First, I created a method to store the bounding box of whatever Mr. Skellington landed on, then adjust his position to right above the top collider. After negating gravity, I finally got him to move at the same rate as the platform. In hindsight, I think I was over-thinking it the whole time, which made the process a lot harder than it needed to be. Overall, I'm pleased with how it turned out.
